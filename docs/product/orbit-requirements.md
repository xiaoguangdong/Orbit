# Orbit / 星筹 需求文档

## 1. 文档信息

- 产品名称：Orbit
- 中文名称：星筹
- 平台：Android
- 文档类型：MVP 需求文档
- 参考来源：`images/` 目录中的滴答清单安卓截图，以及习惯养成打卡 App 的通用用户行为。

## 2. 背景

用户需要一个轻量、清晰、反馈及时的习惯养成工具。现有清单类工具可以管理任务，但习惯坚持需要更强的连续反馈、周期规则、补卡、跳过、统计和复盘能力。Orbit 以习惯打卡为主线，把每日行为组织成「轨道」，用星筹值、连续天数、热力图和里程碑帮助用户长期坚持。

## 3. 目标与非目标

### 3.1 产品目标

- 帮助用户创建并坚持长期习惯。
- 让用户每天能快速完成打卡。
- 让用户清楚看到连续天数、完成率和历史表现。
- 支持基础提醒，减少遗忘。
- 支持轻量设置和个性化分组。
- 支持四象限任务，帮助用户完成当天高优先级事项。

### 3.2 MVP 非目标

- 不做完整项目管理工具。
- 不做复杂社交社区。
- 不做会员付费闭环。
- 不做多端实时同步。
- 不做复杂番茄钟或深度日历系统。

## 4. 用户故事

- 作为用户，我希望快速创建一个每天阅读的习惯，这样我可以从今天开始打卡。
- 作为用户，我希望打开 App 就看到今天未完成的习惯，这样我不用思考下一步。
- 作为用户，我希望完成习惯时只点一下就能打卡，这样不会被记录过程打断。
- 作为用户，我希望看到连续天数和完成率，这样我知道自己坚持得怎么样。
- 作为用户，我希望忘记打卡后可以补卡，这样真实完成的行为不会丢失。
- 作为用户，我希望某天特殊情况可以跳过，这样不会把合理休息视为失败。
- 作为用户，我希望设置提醒时间，这样关键习惯不会被遗忘。
- 作为用户，我希望把习惯按早上、下午、晚上分组，这样每天查看更自然。
- 作为用户，我希望把临时事项放进四象限，这样我可以清楚判断优先级。

## 5. 功能需求

### 5.1 习惯管理

#### 5.1.1 创建习惯

优先级：P0

用户可以创建一个新习惯。

必填字段：

- 习惯名称
- 频率
- 时间段

可选字段：

- 图标
- 颜色
- 目标量
- 提醒
- 开始日期
- 结束日期
- 备注
- 分组

验收标准：

- 未输入名称时不能保存。
- 默认频率为每天。
- 默认时间段为任意时间。
- 创建成功后，新习惯出现在 Today 和 Orbit 页面。
- 如果今天属于计划日，创建后应立即出现在今日列表。

#### 5.1.2 编辑习惯

优先级：P0

用户可以修改习惯字段。

验收标准：

- 修改名称、图标、颜色后，所有列表同步更新。
- 修改频率后，未来计划按新规则生成。
- 已产生的历史打卡记录不被删除。

#### 5.1.3 归档习惯

优先级：P0

用户可以将不再坚持的习惯归档。

验收标准：

- 归档习惯不出现在 Today 的未完成列表。
- 归档习惯保留历史记录。
- 用户可以在 Archived / 已归档中查看并恢复。

#### 5.1.4 删除习惯

优先级：P1

用户可以删除习惯。

验收标准：

- 删除前必须二次确认。
- 删除后习惯与其打卡记录从本地数据中移除。
- 删除操作不可撤销时必须明确提示。

### 5.2 今日打卡

#### 5.2.1 今日列表

优先级：P0

Today 页面展示指定日期的计划习惯。

分组规则：

- Morning / 早上
- Afternoon / 下午
- Evening / 晚上
- Anytime / 任意时间
- Completed / 已打卡

验收标准：

- 默认展示今天。
- 用户可以横向切换最近 7 天。
- 未完成习惯显示在对应时间段。
- 已完成习惯默认折叠或置于已打卡分组。
- 空分组不展示，除非用户打开显示空分组设置。
- Today 页面提供更多菜单，至少包括隐藏详情、隐藏已完成、切换排序方式、显示或隐藏日期条。

#### 5.2.2 快速打卡

优先级：P0

用户点击习惯卡片状态按钮即可完成打卡。

验收标准：

- 点击后状态立即变为已打卡。
- 卡片移动到已打卡分组或显示完成态。
- 连续天数、完成次数、今日完成度同步更新。
- 打卡记录保存时间戳。

#### 5.2.3 取消打卡

优先级：P0

用户可以撤销当天打卡。

验收标准：

- 撤销后状态回到未打卡。
- 连续天数和完成率重新计算。
- 历史记录保留操作日志或直接删除该日完成记录，MVP 可采用直接删除。

#### 5.2.4 跳过

优先级：P1

用户可以将某天习惯标记为跳过。

验收标准：

- 跳过状态不计为完成。
- 默认不中断连续天数。
- 统计中单独显示跳过次数。

#### 5.2.5 补卡

优先级：P1

用户可以为过去日期补打卡。

验收标准：

- 只能补计划日。
- 补卡记录带有补卡标记。
- 补卡后重新计算连续天数、完成率和热力图。
- 历史记录中可识别补卡来源。

### 5.3 频率规则

优先级：P0

支持以下频率：

- 每天
- 每周指定星期
- 每周 N 次
- 每月指定日期
- 自定义间隔，例如每 2 天一次

验收标准：

- 系统能根据频率判断某日期是否需要打卡。
- 非计划日不显示在 Today 的未完成列表。
- 非计划日不影响连续天数。

### 5.4 目标量

优先级：P1

支持不同类型的目标量。

类型：

- 无目标：点击即完成。
- 次数：例如俯卧撑 20 次。
- 分钟：例如阅读 20 分钟。
- 数量：例如喝水 8 杯。
- 距离：例如跑步 3 km。

验收标准：

- 有目标量的习惯打卡时可输入完成值。
- 达到目标值才默认标记完成。
- 用户可以手动确认完成。

### 5.5 提醒通知

优先级：P0

用户可以为习惯设置一个或多个提醒。

验收标准：

- 到提醒时间时发送系统通知。
- 点击通知进入对应习惯或 Today 页面。
- 已打卡后当天不再发送该习惯后续提醒，除非用户设置为持续提醒。
- 用户可在设置中关闭所有提醒。

### 5.6 习惯详情

优先级：P0

习惯详情页展示该习惯的历史与设置。

内容：

- 习惯名称、图标、颜色
- 今日状态
- 当前连续天数
- 最长连续天数
- 总完成次数
- 近 30 天完成率
- 月历记录
- 最近打卡记录

验收标准：

- 数据与 Today 页面一致。
- 用户可以从详情页打卡、撤销、跳过、编辑、归档。

### 5.7 统计洞察

优先级：P1

Insights 页面展示全局统计。

MVP 指标：

- 今日完成率
- 本周完成率
- 当前连续榜
- 最长连续榜
- 近 30 天完成热力图
- 总星筹值

验收标准：

- 指标根据本地记录实时计算。
- 没有数据时展示空状态和创建习惯入口。

### 5.8 里程碑

优先级：P2

Milestones 页面支持目标日与纪念日。

MVP 可先支持基础倒数日：

- 名称
- 日期
- 类型：倒数 / 已过
- 图标
- 提醒

后续支持关联习惯目标。

### 5.9 设置

优先级：P1

设置项：

- 功能模块开关
- 外观主题
- 一周开始日
- 时间制式
- 打卡声音
- 通知总开关
- 分组管理
- 数据导出

验收标准：

- 关闭模块后底部导航或入口隐藏。
- 修改一周开始日后日历与统计同步更新。
- 关闭通知后不再发送提醒。

### 5.10 四象限任务

#### 5.10.1 四象限看板

优先级：P1

Today 页面支持切换到 Quadrants / 四象限任务视图。

象限：

- I 重要且紧急
- II 重要不紧急
- III 不重要但紧急
- IV 不重要不紧急

验收标准：

- 用户可以在 Today 页面在 Habits 和 Quadrants 间切换。
- Quadrants 页面以 2 x 2 布局展示四个象限。
- 每个象限只展示属于该象限的任务。
- 页面显示 Active Tasks 和 Completed Tasks 概览。
- 任务列表按未完成优先和截止状态排序。

#### 5.10.2 创建与编辑任务

优先级：P1

用户可以创建和编辑一个四象限任务。

字段：

- 标题：必填
- 描述：可选
- 象限：必填
- 截止日期：可选
- 截止状态：系统根据当前日期自动计算

验收标准：

- 未输入标题时不能保存。
- 新任务保存后立即进入对应象限。
- 修改象限后任务应立即移动到新的象限。
- 任务需显示 `Overdue / Today / Upcoming / Someday` 中的一种截止状态。

#### 5.10.3 完成、恢复与删除任务

优先级：P1

用户可以完成、恢复和删除任务。

验收标准：

- 完成后任务状态立即更新。
- 若全局开启隐藏已完成，已完成任务不再显示。
- 恢复后任务重新回到原象限。
- 删除后任务从本地数据中移除。

## 7. 关键计算规则

### 7.1 是否计划日

系统根据 Habit.frequencyRule 判断某日期是否为计划日。只有计划日参与未完成、失败、连续天数和完成率计算。

### 7.2 当前连续天数

从最近一个计划日向前计算：

- completed 计入连续。
- skipped 默认不打断，但不增加连续。
- failed 或缺失记录打断连续。
- 非计划日忽略。

### 7.3 完成率

完成率 = completed 计划日数量 / 应完成计划日数量。

skipped 是否计入分母由设置决定，MVP 默认计入分母但单独展示。

### 7.4 星筹值

MVP 规则：

- 普通完成：+10
- 带目标量完成：+12
- 补卡完成：+3
- 跳过：+0
- 失败：+0
- 连续 7 天奖励：+30
- 连续 30 天奖励：+120

## 9. 非功能需求

### 9.1 性能

- Today 页面冷启动后 1 秒内可交互。
- 本地 1000 个习惯、100000 条打卡记录下，Today 页面加载不超过 500 ms。
- 月历和统计计算应缓存或增量计算。

### 9.2 离线

- MVP 必须完整支持离线使用。
- 打卡、编辑、统计均使用本地数据。
- 登录、注册、会话和同步只保留 DTO 与 Repository contract 占位，不接真实服务。

### 9.3 数据可靠性

- 打卡记录写入必须原子化。
- 删除习惯前必须确认。
- App 重启后不丢失当天状态。

### 9.4 隐私

- MVP 不上传个人数据。
- 若后续增加云同步，需要提供隐私说明和同步开关。

### 9.5 无障碍

- 关键按钮提供 contentDescription。
- 打卡状态不能只依赖颜色表达。
- 支持系统字体缩放。

- habit_created
- habit_checked_in
- habit_skipped
- habit_backfilled
- reminder_enabled
- reminder_triggered
- habit_archived
- insights_viewed
- task_created
- task_completed
- task_deleted

关键指标：

- D1 / D7 留存
- 人均创建习惯数
- 今日打卡完成率
- 连续 7 天用户比例
- 提醒开启率
- 补卡使用率

## 11. 验收标准

MVP 完成标准：

- 用户可以创建 3 个不同频率的习惯。
- 用户可以在 Today 页面完成当天打卡。
- 用户可以创建、完成、编辑、删除四象限任务。
- 用户可以查看连续天数和近 30 天记录。
- 用户可以为习惯设置提醒。
- 用户可以归档并恢复习惯。
- 关闭 App 后重新打开，所有状态保持正确。
- 无网络环境下核心功能可用。

## 12. 风险与待确认

- 是否要支持完整待办功能：建议 MVP 不支持，仅保留一次性快速事项。
- 跳过是否影响连续：建议默认不影响，但提供设置。
- 补卡是否限制时间范围：建议 MVP 允许补最近 30 天。
- 星筹值是否影响等级：建议 MVP 先展示数值，等级与徽章后续完善。
- 是否支持桌面组件：用户价值高，但实现成本较高，建议 V1.1。
